# Grill results — 8 design-hole topics

Self-grill (10-round Q&A each) run 2026-06-30, recovered from an archived Discord/piscord session transcript after the live channel was reset. Design-only notes, not a build plan.

---

## 1 — two UI modes vs partial override

1. **Q:** Does per-type override (`fn InvoicePage(obj, meta)` shadows the generic form) actually exist as a mechanism today, or is it proposed? **A:** `GenericUi.cs` already resolves `sys.resolve` by name, app-scope-shadows-lib-scope — the plumbing exists for components generally; it's not yet wired specifically as "override this type's default form."
2. **Q:** So what's the actual gap? **A:** Routing (`instance.app`'s implicit `/Type/:id` → generic `<ObjectForm>`) is hardcoded, not name-resolved through the same override lookup the rest of the lib uses.
3. **Q:** Minimal fix? **A:** Router looks up `fn <TypeName>Page` in app scope first, falls back to `<ObjectForm>` from lib — one lookup, no new config surface.
4. **Q:** Does this cover "override just one field, keep the rest generic" (K2's actual customization unit)? **A:** No — a full page override throws away the generic form's other fields too. Field-level override needs a second seam inside `<ObjectForm>` itself (per-field slot override), not just page-level.
5. **Q:** Is field-level override the same mechanism, one level down? **A:** Should be — `<ObjectForm>` would need to check `sys.resolve("<Type><Field>Field")` per field before falling back to its own default renderer. Not built.
6. **Q:** Config matrix risk — does this stay flat as override surfaces multiply (page, field, per-role, per-viewer)? **A:** Each axis is a separate name-lookup keyed on a naming convention (`TypeName`, `TypeNameField`), not a config table — stays flat as long as we resist a generic "rules engine" and just add more naming conventions.
7. **Q:** Where would it break? **A:** Two types sharing a field name that both want different field overrides on DIFFERENT types — naming convention must include the type, not just the field (`InvoiceAmountField`, not `AmountField`) — already true in the proposal, but worth stating as the actual constraint, not assumed.
8. **Q:** Per-user override (the actual K2 ask) — does this design leave room? **A:** Yes in principle — the resolve lookup would key on `(name, viewer)` later, same shape, just a richer name/context; nothing here forecloses it.
9. **Q:** Cost of NOT building field-level override now? **A:** ERP template can only do whole-page swaps for now — acceptable for MVP scope (per AGENTS.md, M12 designer is deferred anyway), but should be flagged as a known gap in the ERP template doc, not silently absent.
10. **Q:** Verdict? **A:** Page-level override is a one-router-lookup addition, safe to build. Field-level override is the same idea one layer deeper, not yet built, and should wait until a real ERP screen actually needs it (YAGNI holds until it doesn't).

## 2 — invariants vs the access floor

1. **Q:** Is the actor/integrity split actually clean? **A:** Yes for the common case — AccessFloor asks "may THIS actor write," invariant asks "is the RESULT valid" — orthogonal questions, no overlap in the current single-object case.
2. **Q:** Where would they need to see each other's state? **A:** A rule like "only Admin can force a negative balance" — that's an invariant whose truth depends on `currentUser`. Today `invariant` is `{obj}`-scoped only, no access to `currentUser` by design (kept separate deliberately).
3. **Q:** Is that a real gap or a good guard-rail? **A:** Good guard-rail — the moment an invariant reads `currentUser`, it stops being "integrity regardless of who wrote it" and becomes an authorization rule in disguise, defeating the reason for the split. If that case shows up, it belongs in AccessFloor as a conditional floor, not in the invariant.
4. **Q:** Cross-object invariants (sub-ledger reconciles to control account) — does the current design handle these? **A:** No — invariant is defined ON A TYPE, evaluated over one committing object's staged fields. A reconciliation invariant needs to read a DIFFERENT type's aggregate.
5. **Q:** Is that expressible today at all? **A:** Only if the invariant expression can call into a computed value that folds over another extent (`sum(db.subledgerLines where …)`) — computed values already do cross-object folds, so the invariant's boolean expression COULD embed one; the sweep mechanism itself doesn't need to change, just the expression's reach.
6. **Q:** So is cross-object actually solved, or does it just look solved? **A:** Looks solved on paper — untested. A fold over potentially the WHOLE ledger, re-run on every single commit to ANY line, is the real cost — ties directly into topic 4 (computed-not-stored doesn't scale to whole-history folds).
7. **Q:** Does that make invariants unusable for real accounting integrity? **A:** Not unusable — just needs the same ceiling-marking as topic 4: fine for a small/bounded extent (e.g., "this journal entry's own lines sum to zero"), NOT fine yet for "every account in the ledger reconciles on every write."
8. **Q:** What breaks first in practice? **A:** An invariant that's cheap when written, expensive as data grows — a correctness feature that becomes a performance cliff with no warning, the worst kind of hidden ceiling.
9. **Q:** Minimal guard against that? **A:** Document the constraint explicitly wherever `invariant` ships: "scoped to bounded state, cross-object folds are a known future cost" — not a code change, a written ceiling (ponytail: mark the ceiling, don't silently ship it).
10. **Q:** Verdict? **A:** Actor/integrity split is clean and should stay clean (no `currentUser` in invariants, ever). Cross-object invariants are expressible today by composition with computed values, but inherit computed-value's unscaled-fold problem — same open ceiling as topic 4, not a new one.

## 3 — temporal / immutable posted entries

1. **Q:** Does "immutable once posted" need pillar-4's full commit-log/DAG machinery, or something smaller? **A:** Smaller — "can't edit a posted entry" is a TRANSITION GUARD (status field: draft → posted is a one-way gate), not a versioned-history feature.
2. **Q:** Is a transition guard buildable with what exists today? **A:** Close — `invariant`-style commit-time check has access to STAGED vs the loaded base (pre-image) per `CodeExecutor`'s ctx machinery; a guard of the shape "if base.status == Posted then staged.status == Posted and no other field changed" is expressible without any new storage capability.
3. **Q:** Does that actually stop an edit, or just stop a NAIVE edit? **A:** Stops any edit through the normal write path (handler/WS commit) — doesn't stop someone editing the JSON file directly on disk, but that's already true of every other invariant (out of scope, same trust boundary as the rest of storage).
4. **Q:** Does "corrections are reversing entries, never edits" fall out of the guard for free? **A:** No — the guard only PREVENTS editing a posted entry. It doesn't provide the "create a reversing entry" workflow; that's ordinary app logic (a handler that creates a new entry with negated amounts), not a platform feature.
5. **Q:** So what's actually missing from "MVP-era ERP template"? **A:** Just the guard + a documented app-level convention (a `Handler` for "reverse this entry" that stages a new object). No platform work beyond the transition-guard invariant shape from topic 2.
6. **Q:** Does the MVP need audit-trail-of-WHO-posted-it, or just can't-edit-once-posted? **A:** The seed conflates them — "audit trail" implies knowing who/when, which needs at minimum a stamped field (`postedBy`, `postedAt`), not history. Cheap, already buildable (ordinary fields set inside the posting handler).
7. **Q:** Does full pillar-4 (view the ledger as of a past moment) matter for MVP-era ERP? **A:** Not for a first dogfood pass — "can't tamper with a posted entry" satisfies the integrity story; "replay the ledger as it looked last Tuesday" is a real accounting need eventually but not gate #3 (dogfood one app).
8. **Q:** Risk of shipping the cheap version? **A:** If the ERP template's docs oversell it as "full audit history," that's a false claim. Must be labeled precisely: transition-guard immutability + stamped fields now, versioned time-travel later (M13+).
9. **Q:** Does this cheap version block or complicate the eventual pillar-4 build? **A:** No — a status-guard invariant is orthogonal to an append-only commit-log; pillar-4 wraps ALL objects uniformly later, doesn't need the guard removed first.
10. **Q:** Verdict? **A:** MVP needs the transition-guard invariant (buildable now, small) + stamped fields (already buildable) + a reversing-entry handler pattern (app-level, no platform change). Full pillar-4 stays parked, correctly, per the existing DECISIONS.md memo.

## 4 — computed-not-stored vs giant aggregates

1. **Q:** Does "computed not stored" literally mean re-summing the whole history on every read? **A:** Yes, as designed today — a computed value is a pure fold over an extent, re-evaluated whenever accessed; no incremental/materialized state exists.
2. **Q:** Is the memo cache (per-render, structural) a mitigation? **A:** Only within ONE render pass — it dedupes repeated reads of the same computed value in a single request, doesn't persist across requests or amortize the fold itself.
3. **Q:** At what scale does this actually hurt? **A:** A fold over an in-memory JSON-file-backed extent of, say, 100k postings, done on every page load that shows a balance — real latency, not theoretical, well within ERP's expected data volumes.
4. **Q:** Does the current storage model (`JsonFileInstanceStore`, full-file rewrite) make this worse? **A:** Yes — the WHOLE store loads into memory regardless; a fold over history is at least an in-memory scan (fast per-item, but O(n) with no shortcut), not a disk-bound query — better than it sounds, still doesn't scale past memory-resident data.
5. **Q:** Is "computed not stored" a wrong principle, or an incomplete one? **A:** Incomplete, not wrong — it's correct for FRESH, ungrown data (M1-M11 test scale); ERP-scale history is the first workload that actually stresses it.
6. **Q:** What primitive is missing? **A:** A running-total / snapshot primitive — e.g., a periodically materialized checkpoint (account balance as of end-of-month) plus a fold over only the DELTA since — doesn't exist in any form today.
7. **Q:** Can this be retrofitted without breaking pillar-2 (computed never stored)? **A:** Only if the checkpoint is framed as a CACHE (derivable, rebuildable, never authoritative) — same shape pillar-4's differential checkpoints already describe for the commit-log design (DECISIONS.md: "Differential checkpoints... pure functions over already-immutable history"). Reuse that shape rather than inventing a second one.
8. **Q:** Does that mean the aggregate-checkpoint problem and pillar-4's checkpoint design are the SAME problem? **A:** Structurally yes — both need "a cheap current answer derived from a long, append-only history without re-scanning it." Solving pillar-4 properly likely solves this for free later.
9. **Q:** So is this genuinely blocking the MVP, or a known-but-deferrable ceiling? **A:** Deferrable for gate #3 (one dogfood app, `devlog` — small data). Blocking for a REAL ERP template claiming to handle real transaction volume — must be marked as an explicit, named limitation in the ERP domain doc, not silently implied as solved.
10. **Q:** Verdict? **A:** "Computed not stored" holds for MVP-scale data; breaks for whole-history aggregates at ERP scale. The fix (checkpoint/differential-fold primitive) doesn't exist yet and rides on the same design as pillar-4's commit-log checkpoints — don't build it twice, park both together.

## 5 — footprint model vs pagination

1. **Q:** Does render-as-planner assume the whole view is always small? **A:** Implicitly yes — the model is "render the URL, harvest whatever got touched"; if the render touches 100k rows to produce a table, the footprint IS 100k rows, no smaller.
2. **Q:** Is that a flaw in the harvest mechanism, or in the RENDER (the app code) itself? **A:** The render — `foreach db.invoices` over 100k rows was always going to be slow/huge regardless of footprint harvesting; the harvest just faithfully reports what the render actually touched.
3. **Q:** So is pagination a footprint-model problem at all? **A:** No — it's a rendering-primitive gap: there's no `Take`/`Skip`/paged `foreach` construct in Code today (confirmed: no pagination primitive found in `ExecValues.cs`/`CodeExecutor.cs`).
4. **Q:** If a paged `foreach` existed, would the footprint model handle it for free? **A:** Yes — the footprint is defined as "what the render actually accessed" (structural privacy, already how it works); a render that only iterates page 3's 20 rows would only harvest those 20, no model change needed.
5. **Q:** What about `Table/grid design` — does the existing generic `SetTable` already limit rows? **A:** No — per memory, `SetTable` shows ALL non-collection props as columns for the WHOLE set; no row-limiting exists there either.
6. **Q:** Where's the actual seam, precisely? **A:** In Code's collection iteration primitives (`foreach`, and any `SetTable`-style generic component) — needs a paged variant (`take(n, offset)` or similar) that view-state can key its "which page am I on" through (already fits — view-state ships arbitrary scalars, a `page` int is nothing new).
7. **Q:** Minimal fix? **A:** Add one collection primitive (`.page(n, size)` or similar), have `SetTable` accept a page prop from view-state, done — no change to WS/footprint/harvest machinery at all.
8. **Q:** Does this interact with topic 4 (giant aggregates)? **A:** Only tangentially — pagination limits what's RENDERED; an aggregate (a balance) still folds the WHOLE set regardless of which page is shown. Separate ceilings, don't conflate them.
9. **Q:** Risk of NOT building this before ERP dogfooding? **A:** `devlog` (the actual MVP dogfood target) is small — this is a real ERP-template gap, not a gate-#3 blocker. Same "named limitation, not silent" treatment as topic 4.
10. **Q:** Verdict? **A:** Footprint model itself is fine and needs no change — the actual gap is a missing paged-collection primitive in Code. Cheap fix, deferred correctly since it's not required for the current MVP dogfood target.

## 6 — design-time permission grain

1. **Q:** Is reaching the designer gated by the same role ruleset as app data? **A:** No — confirmed by code: `AccessFloor` gates apply per-type/per-field to an APP'S OWN schema; the designer instance (`instances/1`) has no `floor`/`currentUser` rules in its own `app.deenv` at all.
2. **Q:** So what gates designer access today? **A:** Nothing, at the per-user level — whoever can reach the kernel host's URL for that instance can use it. It's effectively single-operator, matching AGENTS.md's stated Stage-1 scope.
3. **Q:** Is that a bug, or just unbuilt-yet scope? **A:** Unbuilt-yet — AGENTS.md explicitly defers M12 (visual component designer) and multi-operator concerns; this isn't a regression, it's the honest current state.
4. **Q:** What would "regular user can extend their own app" actually require? **A:** The designer itself would need to become an app WITH access rules — i.e., apply the SAME `AccessFloor` mechanism to the designer's own types (`Design`, `DesignField`, etc.) that any other app already uses. No new mechanism, just turning the existing one on for instance 1.
5. **Q:** Is that as simple as it sounds? **A:** Mostly — but "my own app" implies a per-user OWNERSHIP concept (which designs belong to which user), which doesn't exist in the designer's schema today (per memory: designer stores `db.instances`, no per-owner field).
6. **Q:** Does adding an `owner: User` field + a floor rule (`currentUser == obj.owner`) close this? **A:** Structurally yes — same primitive as every other per-user data-ownership rule in the system; no new kernel-level concept required.
7. **Q:** Does self-service schema-editing raise a NEW risk beyond data access? **A:** Yes — editing schema is qualitatively different from editing data: a bad schema edit can break OTHER users' data on a shared instance (kernel hosts N instances, but if two users design in the SAME instance, their schemas collide). Needs "instance per designing user" or a much bigger multi-tenant schema story — likely M13+ territory (schema versioning), not solvable by an access rule alone.
8. **Q:** So is "regular user self-service" mostly an access-rule problem or mostly a schema-isolation problem? **A:** Schema isolation is the harder, bigger one — access rules gate WHO can reach the designer; isolation determines whether two users' edits can coexist at all.
9. **Q:** Does today's per-instance model (kernel hosts N instances) already give isolation "for free" if each user gets their OWN instance? **A:** Yes — that sidesteps the schema-collision problem entirely (ties directly to topic 7's SaaS-instance-per-customer axis) at the cost of N separate instances instead of one shared designer.
10. **Q:** Verdict? **A:** Today: all-or-nothing, single-operator, by omission not by design choice. Near-term close: apply existing AccessFloor + an ownership field to gate WHO reaches the designer. Real "self-service, shared" story needs schema isolation across users, which the instance-per-user model already sidesteps — worth deciding which axis the ERP template actually wants before building either.

## 7 — single-instance-many-users vs instance-per-customer

1. **Q:** Does the current model support many users sharing ONE instance today? **A:** Yes — confirmed in `ws.ts`: each WS connection has its own `clientId`, its own session-bound `login`/`logout`, its own `currentUser` — many concurrent sessions against one instance is the existing architecture (memory: "sessions are users").
2. **Q:** Does per-user CUSTOMIZATION (K2-style) already ride on this? **A:** The plumbing (per-session identity) is there; the actual FEATURE (per-user saved view/menu prefs) doesn't exist — no per-user preference object type ships today (confirmed earlier: no menu/nav object, no per-user UI state).
3. **Q:** What would per-user customization need, concretely? **A:** An ordinary app-level type (`UserPreference` or similar, `owner: User`) plus reads gated by `currentUser == obj.owner` — same primitive as everything else, no platform work.
4. **Q:** Does SaaS (instance-per-customer) conflict with single-instance-many-users? **A:** No — they're not mutually exclusive; kernel already hosts N instances, and EACH instance can independently be single-user or many-user. The axes are orthogonal, not a fork in the road.
5. **Q:** So which axis does the ERP TEMPLATE actually need? **A:** Many-users-one-instance — an ERP is inherently one company's shared ledger/data with many employees, not one instance per employee. SaaS (instance-per-tenant-COMPANY) is a hosting decision layered on top, unrelated to the ERP template's own data model.
6. **Q:** Does Stage-1 "single-operator MVP" contradict needing many-users-one-instance? **A:** No — "single-operator" (per STAGES.md) describes who RUNS the deenv install (one person deploys/owns the box), not how many end-users log into ONE hosted app. Different axis entirely — worth flagging since the wording invites exactly this confusion.
7. **Q:** Is there a real gap once many real users hit one instance concurrently? **A:** Concurrency itself — ties directly to topic 8 (write races) and topic 6 (per-user data ownership/gating), not a NEW gap unique to this topic.
8. **Q:** Does the render/memo-cache correctly key on viewer, not just args, for per-user views? **A:** Flagged earlier as a "one-way door" to watch (`(slot, args, viewer)` not `(slot, args)`) — not verified against actual `ExecValues.cs`/memo-key code in this pass; worth a dedicated grounded check before shipping any per-user view feature, not assumed safe.
9. **Q:** Risk of skipping that verification? **A:** If the memo key doesn't include viewer, one user's per-user customized view could leak into another user's cached render — a real correctness bug, not cosmetic.
10. **Q:** Verdict? **A:** Model already supports many-users-one-instance (sessions=users, already built). ERP wants this axis, not instance-per-customer (that's a hosting concern, layered independently). Real open item: verify the memo cache actually keys on viewer before any per-user customization ships — flagged, not yet confirmed.

## 8 — write concurrency / number sequences

1. **Q:** Is there an allocation primitive for sequence numbers today? **A:** No — confirmed, nothing resembling a counter/sequence primitive exists in Code or storage.
2. **Q:** Does the commit path serialize writes enough to make this a non-issue anyway? **A:** Partially — `JsonFileInstanceStore` takes a `lock (_sync)` around the actual Apply/Save (confirmed, `JsonFileInstanceStore.cs`), so the FINAL write-to-disk step is atomic and serialized.
3. **Q:** Does that lock cover the DECISION (computing "next number = count + 1"), or just the write? **A:** Just the write — `WsHandler.HandleCommit` builds the whole changeset (including any computed "next number" value baked into a staged field) BEFORE calling `_store.CommitBatch(...)`, which is where the lock actually starts (confirmed at `WsHandler.cs:746`, comment: "everything validated — mint + link + write... in one lock").
4. **Q:** So is the race real? **A:** Yes, if a handler computes a sequence number via a READ (e.g., `count(db.orders) + 1`) during its OWN execution, before the locked apply — two concurrent handler runs (two browser tabs creating an order at the same instant) can both read the same count and both stage the same "next" number.
5. **Q:** Does anything upstream of the lock ALSO serialize handler execution itself (not just the final write)? **A:** Not found in this pass — no semaphore/queue around handler execution was found; each WS connection appears to run its handler independently until it reaches the shared store lock. Should be verified against the actual WS message-processing loop before treating this as fully certain.
6. **Q:** Assuming the race is real, what's the actual failure mode? **A:** Two orders silently sharing the same "SO/2026/0001" number — not a crash, a silent DATA INTEGRITY bug, the worst kind (matches the exact class of problem `invariant` (topic 2) exists to catch).
7. **Q:** Does an invariant (topic 2) fix this, or just detect it? **A:** Detects, doesn't prevent — a `invariant uniqueNumber = ...` would reject the SECOND commit that collides, turning silent duplication into a visible failure (good), but the user still needs a retry path (re-run the handler, get a fresh number) — not automatic today.
8. **Q:** Is there a cheaper real fix than "add a full sequence-allocator primitive"? **A:** Yes — since the STORE-level lock already exists and already serializes the final apply, move the number computation INSIDE that locked section (i.e., let `CommitBatch` compute "next number" itself, atomically, rather than the handler pre-computing it) — reuses the existing lock, no new primitive.
9. **Q:** Does that fit the existing model without a new abstraction? **A:** Yes — this is exactly a `computed-at-commit` value, structurally close to how `invariant` (topic 2) already proposes evaluating something INSIDE the commit sweep, right before persisting — same seam, different job (assign, not just check).
10. **Q:** Verdict? **A:** The race is real today (or at minimum unverified-safe) because the lock protects the WRITE, not the DECISION. No new allocator primitive needed — the fix is doing the number-assignment INSIDE the existing store lock (same commit-sweep seam as `invariant`), not before it. Needs an actual concurrent-request test to confirm before calling it fixed either way.

---

Common thread across all 8: none of these are new/foreign problems — every fix reuses a mechanism that already exists (the commit-sweep seam, the resolve-by-name lookup, the existing lock, the existing session/viewer identity). The gaps are real but narrow. Biggest genuinely-unverified claim: #8's race and #7's memo-cache-viewer-key — both worth an actual test before either gets marked solved or dismissed.
